# AGENTS.md

> 이 파일은 **AI 코딩 에이전트(Google Antigravity, Claude 등)가 이 저장소에서 작업하기 전에 먼저 읽는 규칙집**입니다.
> 사람 신입 개발자에게 주는 온보딩 문서를 AI용으로 만든 것이라고 생각하면 됩니다.
> 여기 적힌 내용은 이 프로젝트의 **큰 방향**과 **에이전트의 작업/응답 태도**를 정의합니다.

---

## 📌 이 프로젝트는 어떤 프로젝트인가요?

- **포인핸드(Pawinhand) 클론코딩 + AI 매칭 + 지도 시각화** 서비스입니다.
- 유기동물 입양 희망자가 **AI로 "내가 입양해도 될지"를 진단하고, 나에게 맞는 아이를 추천받는** 웹 서비스입니다.
- **비개발자와 약간의 개발 경험이 있는 팀원이 섞인 4인 팀**이, **4일 안에** 동작하는 MVP 데모를 만드는 것이 목표입니다.
- 완벽한 제품이 목표가 **아닙니다**. **핵심 기능이 실제로 돌아가는 데모**를 빠르게 만드는 것이 목표입니다.
- **항상 팀에 코딩을 잘 모르는 사람이 있다고 전제하세요.** 전문 용어를 쓸 때는 반드시 쉬운 말로 한 번 더 풀어 설명하고, "이 코드가 뭘 하는지 / 왜 이렇게 하는지"를 매번 함께 설명합니다. 코드만 던지고 끝내지 않습니다.

## 🎯 만들려는 핵심 기능 (우선순위 순)

에이전트는 아래 우선순위를 기준으로 작업합니다. **Must → Should → Could** 순서를 지키고, Must가 다 돌아가기 전에는 Could에 손대지 않습니다.

### Must (없으면 서비스가 성립 안 됨)

1. **입양 적합 판단 AI** — 사용자에게 6개 항목을 입력받아 "입양 가능 / 조건부 가능 / 신중히 재고" 3단계 등급 + 보완점을 판단.
2. **나와 맞는 유기동물 매칭 AI** — 1단계 입력값을 그대로 활용해 보호동물 3~5마리 추천 (각 카드에 추천 이유 한 줄 + 매칭 점수 1~10점 표시).
3. **보호동물 상세 페이지** — 매칭 카드를 클릭하면 도달하는 화면. 수집한 동물 정보 필드를 표시.
4. **보호소 질문지 자동 생성** — 입양 문의 전 보호소에 물어볼 질문 목록 (공통 템플릿 우선).

### Should (있으면 좋고 비교적 빨리 됨)

- 입양 준비 체크리스트, 입양 문의/절차 안내, 보호소 질문지의 AI 커스텀 질문.

### Could (시간 남으면)

- 보호동물 리스트 + 검색/필터, **지도 시각화 + 클러스터링**, 가이드북(정적 콘텐츠).

### Won't (이번 MVP 밖 — 방향성만)

- RAG 챗봇, 펫보험·상품 연계(BM), 가족 공유용 요약 카드.

> **핵심 원칙**: 검색·지도(탐색)는 진입 통로일 뿐이고, 서비스의 진짜 가치는 "입양해도 될까 → 어떤 아이 → 이 아이로 결정"의 판단을 돕는 데 있습니다. **진단형 단일 흐름**에 집중합니다.

## 🖥 화면(페이지) 목록

에이전트가 만들어야 하는 화면은 아래 8개입니다. 괄호 안은 우선순위입니다.

| # | 화면 | 설명 | 우선순위 |
|---|---|---|---|
| 1 | **홈 화면** | 서비스 소개 + "적합도 진단 시작" CTA + 유기동물 목록 진입 링크 | Must |
| 2 | **유기동물 전체 목록** | 수집한 보호동물을 카드로 나열 (사진 + 품종·나이·성별·크기·보호소·공고 남은 기간) | Must |
| 3 | **유기동물 상세 페이지** | 동물 한 마리의 전체 정보 + 보호소 질문지로 가는 버튼 | Must |
| 4 | **유기동물 지도 보기** | 보호소 위치를 지도에 핀으로 표시, 가까운 건 클러스터링 | **Could(후순위)** |
| 5 | **입양 적합도 진단 결과** | 3단계 등급 + 잘 맞는 점 + 보완점 + 예상 양육비 + "매칭 보기" CTA | Must |
| 6 | **매칭 결과 목록** | 나에게 맞는 유기동물 3~5마리 추천 (카드마다 추천 이유 한 줄 + 매칭 점수) | Must |
| 7 | **매칭 결과 상세** | #3 상세 페이지와 동일하되, **"왜 이 아이를 추천하는지" 설명 블록만 추가** | Must |
| 8 | **보호소 질문지 페이지** | 카테고리 탭 + 질문 리스트(모든 질문 복사) + 입양 준비 체크리스트 + 보호소 연락처/문의 | Must |

> **입력 화면 참고**: 진단 입력은 **2단계**로 나뉩니다.
> - **1단계 — 나의 생활환경**: 주거형태 · 하루 외출시간 · 산책 가능 시간 · 반려동물 경험 · 월 예산 · 자녀 계획 여부 (6개)
> - **2단계 — 원하는 아이 성향** *(시안 반영)*: 활동성 선호 · 사람 친화도 선호 · 성향 키워드(다중 선택)
> 이 2단계 입력을 마치면 화면 #5(진단 결과)로 넘어갑니다.

> **#3 상세와 #7 매칭 상세의 차이**: 두 화면은 **같은 컴포넌트를 재사용**하고, #7에만 "추천 이유 + 매칭 점수" 블록이 위에 하나 더 붙습니다. 별도 페이지를 새로 만들지 말고 prop(예: `recommendReason`)으로 분기하세요.

## 🧠 AI 로직 상세 (구현 시 참고)

**적합 판단 AI**
- **입력은 2단계로 받습니다.**
  - 1단계(생활환경 6개): 주거형태 · 하루 외출시간 · 산책 가능 시간 · 반려동물 경험 · 월 예산 · 자녀 계획 여부.
  - 2단계(원하는 아이 성향, *시안 반영*): 활동성 선호(3택) · 사람 친화도 선호(3택) · 성향 키워드(다중 선택).
- 출력: 3단계 등급(입양 가능 / 조건부 가능 / 신중히 재고) + 잘 맞는 점 + 보완·확인 필요 사항 + 예상 월 양육비 범위 + 매칭으로 넘어가는 CTA.
- **판단 근거는 룰 점수표가 아니라 LLM 프롬프트 기반(RAG)**. 참고자료(파양 사유 통계, 입양 전제조건 통계, 견종 특성 등)를 프롬프트에 텍스트로 붙여 넣어 판단.

**입력 선택지 (시안 확정본 — 이 값 그대로 구현)**

*1단계 — 나의 생활환경 (6개, 모두 단일 선택)*
| 항목 | 선택지 |
|---|---|
| 주거형태 | 아파트 / 빌라·다세대 / 단독주택 / 원룸·오피스텔 |
| 하루 외출시간 | 4시간 미만 / 4~8시간 / 8시간 이상 |
| 산책 가능 시간 | 30분 미만 / 30분~1시간 / 1시간 이상 |
| 반려동물 경험 | 없음 / 있음(현재는 없음) / 있음(현재도 있음) |
| 월 예산 | 10만원 미만 / 10~20만원 / 20~30만원 / 30만원 이상 |
| 자녀 계획 여부 | 자녀 없음·계획 없음 / 자녀 있음 / 계획 있음 |

*2단계 — 원하는 아이 성향 (시안 반영)*
| 항목 | 형태 | 선택지 |
|---|---|---|
| 활동성 선호 | 단일 선택 | 조용하고 차분한 아이 / 적당히 활발한 아이 / 활발하고 에너지 넘치는 아이 |
| 사람 친화도 선호 | 단일 선택 | 독립적인 편이 좋아요 / 적당히 붙어있는 게 좋아요 / 항상 곁에 있고 싶어요 |
| 성향 키워드 | **다중 선택** | 온순해요 / 독립적이에요 / 사람을 좋아해요 / 산책을 좋아해요 / 다른 동물과 잘 지내요 / 초보자와 잘 맞아요 |

**매칭 AI**
- 사용자 입력(1단계 6개 + 2단계 성향) + 동물 데이터 필드(`activity`·`aggression`·`sociability`·`health_state` 등 척도형 수치, `animal_age`·`animal_sex`·`animal_weight`·`city`·`personality_comment`·`breeds` 등)를 함께 LLM에 넣어 산출.
- 3~5마리 추천, 각 카드에 **추천 이유 한 줄 + 매칭 점수(1~10)** 필수 표시.
- "신중히 재고" 등급이어도 매칭 단계는 진행하되, 낮은 점수는 정직하게 노출.

**데이터 소스**
- APMS/포인핸드에서 **수동 수집한 고정 데이터(JSON/DB)**를 사용. 자동 배치·스케줄러 없음.
- 시연 중 외부 서버 상태에 의존하지 않도록, 데모용 데이터는 사전에 한 번 수집해 고정해둠.

**출력 형식**
- LLM 응답은 **JSON(Structured Outputs)으로 강제**해서 파싱 에러 확률을 낮춤.
- 결과 JSON은 프론트로만 전달하고 **서버에 저장하지 않음**. 이름·연락처·주소 등 식별정보는 수집하지 않음(익명).

> 아래 필드명은 **제안(초안)**입니다. 실제 포인핸드 응답 필드명과 다를 수 있으니, 수집 후 이름이 맞는지 확인하고 필요하면 조정하세요.

### ① 동물 데이터 (Animal) — 고정 데이터

화면 #2(목록), #3/#7(상세), 매칭 AI 입력에 모두 쓰입니다. 한 마리당 아래 필드를 가진 객체로 만들고, 전체를 배열(JSON)로 저장합니다.

|---|---|---|---|
| `id` | string | 동물 고유 ID (공고번호 활용 가능) | 전체 |

| `photos` | string[] | 사진 URL 목록 | 목록·상세 |

| `breeds` | string | 품종 | 목록·상세·매칭 |
| `animal_age` | number/string | 나이(개월 또는 추정) | 목록·상세·매칭 |

| `animal_weight` | number | 체중(kg) | 상세·매칭 |
| `size` | string | 크기 (소형/중형/대형) | 목록·필터 |
| `neutered` | boolean/string | 중성화 여부 | 상세 |
| `health_state` | number(1~5) | 건강 상태 척도 | 상세·매칭 |

| `notice_end` | date | 공고 종료일(남은 기간 계산) | 목록·상세 |
| `found_location` | string | 발견 장소 | 상세 |
| `shelter_id` | string | 보호소 ID(아래 Shelter와 연결) | 상세·지도 |
| `adoption_support` | boolean | 입양지원 여부 | 상세 |
| `adoption_support_detail` | string | 입양지원 상세 | 상세 |

> **주의**: 척도 필드(`activity` 등 1~5)는 화면에 숫자로 보여주지 말고 "활동량 보통", "사람을 좋아함"처럼 자연어로 변환해 표시하세요.

### ① 보호소 데이터 (Shelter) — 고정 데이터

화면 #8(질문지 하단 연락처), #4(지도 핀)에 쓰입니다.

| 필드명 | 타입 | 설명 |
|---|---|---|
| `shelter_id` | string | 보호소 고유 ID |
| `name` | string | 보호소 이름 (예: 서울 동물사랑 보호소) |
| `phone` | string | 문의 전화 (예: 02-1234-5678) |
| `hours` | string | 운영 시간 (예: 평일 10:00–18:00) |
| `guide` | string | 상담 안내 문구 |
| `address` | string | 주소 |
| `lat` / `lng` | number | 위도/경도 (지도 핀용, #4에서만 필요) |

### ② 진단 API — `POST /api/diagnose`

**요청(Request)** — 사용자 입력 2단계를 그대로 보냄:
```json
{
  "housing": "원룸·오피스텔",
  "out_hours": "8시간 이상",
  "walk_time": "30분 미만",
  "pet_experience": "없음",
  "budget": "10~20만원",
  "child_plan": "계획 있음",
  "activity_pref": "적당히 활발한 아이",
  "sociability_pref": "적당히 붙어있는 게 좋아요",
  "keywords": ["사람을 좋아해요", "산책을 좋아해요"]
}
```

**응답(Response)** — 화면 #5(진단 결과)를 그리는 데 필요한 값:
```json
{
  "grade": "조건부 가능",
  "good_points": ["재택 시간이 있어 혼자 두는 시간이 짧아요", "..."],
  "check_points": ["원룸이라 중대형견은 어려울 수 있어요. 소형견을 추천해요", "..."],
  "monthly_cost": { "min": 12, "max": 19, "gov_support": 25 }
}
```
> `grade`는 반드시 세 값 중 하나: `"입양 가능"` / `"조건부 가능"` / `"신중히 재고"`. "부적합" 같은 값은 쓰지 않습니다.

### ② 매칭 API — `POST /api/match`

**요청**: 진단 API와 동일한 사용자 입력(위 요청 body 그대로) 전송.

**응답** — 화면 #6(매칭 목록)용. 동물 3~5마리에 추천 이유·점수를 붙여 반환:
```json
{
  "results": [
    {
      "animal_id": "경기-2024-00123",
      "match_score": 8,
      "recommend_reason": "하루 30분 산책이면 충분해서 지금 생활과 잘 맞아요"
    }
  ]
}
```
> 프론트는 `animal_id`로 위 동물 데이터를 찾아 카드를 그립니다. 즉 매칭 응답은 **점수·이유만** 담고, 나머지 표시 정보는 고정 동물 데이터에서 가져오는 구조를 권장합니다(응답을 가볍게 유지).
> 화면 #7(매칭 상세)의 "왜 추천하는지" 블록은 이 `recommend_reason` + `match_score`를 재사용합니다.

### ② 보호소 질문지 API — `POST /api/questions` (또는 정적 템플릿)

- **1차(권장)**: 카테고리별 공통 질문 템플릿을 **백엔드에 고정 데이터로** 둡니다. AI 호출 없이 바로 응답 → 빠르고 토큰 비용 0.
  ```json
  {
    "categories": ["건강상태", "행동/성격", "산책/활동", "사회성", "입양절차"],
    "questions": [
      { "no": 1, "category": "건강상태", "text": "예방접종 기록이 최신 상태이며 증빙이 가능한가요?" }
    ],
    "checklist": ["동물을 키울 수 있는 주거 환경인가요?", "..."]
  }
  ```
- **2차(시간 되면, Should)**: 위 공통 질문 뒤에, 해당 동물의 `personality_comment`·`health_state` 등을 넣어 AI가 1~2개 맞춤 질문을 추가 생성.

### 데이터 취급 원칙 (보안)
- 진단/매칭 **요청·응답은 서버에 저장하지 않습니다.** 요청 처리 후 응답만 돌려주고 끝냅니다.
- 사용자 입력 항목은 개인식별정보가 아니므로 **회원가입 없이 익명**으로 받습니다.
- OpenAI API 키는 `.env`(로컬) / Railway 환경변수(배포)로만 관리하고 Git에 올리지 않습니다.

## 🌿 협업 / Git 규칙 (항상 지킬 것)

> 이 규칙은 **코드를 커밋·푸시할 때마다 항상 적용**됩니다. 기능 종류와 상관없이 예외 없이 따릅니다.

### 브랜치 전략
- **`main`**: 배포·최종 완성본 브랜치. **직접 커밋/푸시 절대 금지.**
- **`feature/...`**: 기능 개발용 독립 브랜치. 모든 작업은 여기서 하고, PR을 거쳐 `main`에 합칩니다.
- **브랜치 생성 필수**: 새로운 개발 작업을 처음 시작할 때는 무조건 브랜치를 새로 따고 시작합니다.
- **단계별 브랜치 순환**: 한 단계 진행할 때마다 새로운 브랜치를 사용하며, 다음 단계 브랜치를 따기 전 기존 브랜치의 작업물은 반드시 **조장의 허락을 받아 원격 push 및 PR(Pull Request)까지 완료**해 둡니다.
- 새 작업 시작 전 항상 최신 `main`을 받은 상태에서 브랜치를 팝니다.
  ```bash
  git checkout main
  git pull origin main
  git checkout -b feature/fe-login   # 새 브랜치 만들고 이동
  ```

### 브랜치 이름 규칙
형식: `접두어/파트-작업내용` (소문자, 단어 구분은 하이픈 `-`)

| 용도 | 형식 | 예시 |
|---|---|---|
| 프론트 기능 | `feature/fe-기능명` | `feature/fe-login`, `feature/fe-main-page` |
| 백엔드 기능 | `feature/be-기능명` | `feature/be-matching-api`, `feature/be-db-setup` |
| 버그 수정 | `fix/fe-버그명` / `fix/be-버그명` | `fix/be-match-format` |

### 커밋 컨벤션
형식: `Type: 요약` (요약은 한글 가능). 제목만 보고 무슨 변화인지 알 수 있게 씁니다.

| 타입 | 설명 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 (README, 가이드 등) |
| `style` | 포맷팅·세미콜론 등 (로직 변경 없음) |
| `refactor` | 기능 변화 없는 구조 개선 |
| `chore` | 빌드·패키지 설정·`.gitignore` 등 |

예: `feat: 유기동물 상세 페이지 UI 구현` / `fix: 백엔드 매칭 API 데이터 포맷 에러 수정`

### 커밋 & 푸시 & PR 흐름
```bash
git status              # 변경 파일 확인
git add .
git commit -m "feat: 유기동물 상세 페이지 UI 구현"
git push origin feature/fe-login   # 내 브랜치로 푸시 (main 아님!)
```
그다음 GitHub에서 `Compare & pull request` → 리뷰어에 **`haneebunny`** 지정 → 작업 내용 요약 작성 → `Create pull request` → **조장(신하니)에게 알립니다.**

### ⚠️ 필수 주의사항
- 🚫 **`main`에 직접 푸시 절대 금지.** 반드시 `feature/` 브랜치 → PR → merge 경로만 사용합니다.
- 📦 **의존성 추가 시 lock 파일을 반드시 커밋에 포함합니다.**
  - 프론트(npm): `package.json` + `package-lock.json`
  - 백엔드(Poetry): `pyproject.toml` + `poetry.lock`
  - (아나콘다 사용자는 pull 받은 뒤 추가된 패키지를 수동 설치해야 합니다.)

## 🗣 에이전트가 답변/작업할 때 꼭 지켜야 할 것

1. **전문 용어 = 쉬운 설명 동반**
   예: "이건 API 엔드포인트예요. 쉽게 말하면 '프론트엔드가 데이터를 요청하는 창구'라고 생각하면 됩니다."
2. **왜(Why)를 먼저, 어떻게(How)는 그다음** — 코드/설정을 제안하기 전에 왜 그렇게 하는지 1~2문장으로 먼저 설명합니다.
3. **단계별로 쪼개서 안내** — 한 번에 다 던지지 말고 "1단계 → 확인 → 2단계" 식으로 진행합니다.
4. **막혔을 때를 대비한 대안 제시** — 에러가 날 법한 부분은 미리 "이런 메시지가 뜨면 이렇게 하세요"라고 짚어줍니다.
5. **시간 제약(4일) 고려** — "완벽한 방법"보다 "지금 가장 빠르고 검증된 방법"을 우선합니다. 과도한 엔지니어링(불필요한 추상화, 과한 테스트 커버리지)은 지양합니다.
6. **역할 분담 고려** — 비개발자에게는 기획/문서화/디자인/테스트를, 개발 경험자에게는 실제 구현을 자연스럽게 제안할 수 있습니다.
